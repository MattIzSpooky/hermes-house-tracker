# Hermes House Tracker

Hermes is a real estate tracking application that scrapes property listings from [Funda.nl](https://www.funda.nl) and persists them for analysis and tracking over time. Users can initiate scraping sessions, search and filter tracked listings (including radius search around their saved home address), view price history charts, re-scrape individual listings on demand, chat with an AI assistant backed by a local LLM, and schedule recurring agent tasks (watches, digests, research) that deliver in-app and email notifications.

## Architecture

The backend is a **modular monolith** built with [Spring Modulith](https://spring.io/projects/spring-modulith). Modules communicate via Spring's application event system — no direct cross-module bean dependencies. This enforces clear boundaries while keeping deployment simple.

### Modules

| Module | Package | Responsibility |
|---|---|---|
| `scraping` | `com.kropholler.dev.hermes.scraping` | Scraping session lifecycle, queue processing, delegates fetching to funda-proxy |
| `listing` | `com.kropholler.dev.hermes.listing` | Listing persistence, deduplication, price history, geocoding, spatial search |
| `report` | `com.kropholler.dev.hermes.report` | Report generation: days tracked, price trend, price change % |
| `ai` | `com.kropholler.dev.hermes.ai` | Ollama-backed LLM: streaming chat with persisted history, listing summaries, AI tools, scheduled agent tasks |
| `favorites` | `com.kropholler.dev.hermes.favorites` | Per-user favorite listings |
| `notification` | `com.kropholler.dev.hermes.notification` | In-app and email notifications generated by agent tasks |
| `profile` | `com.kropholler.dev.hermes.profile` | User home address (geocoded on save; used as a search origin) |

Cross-cutting concerns live outside the module packages proper: `security` (the JWT `Authentication` → user id helper), `config` (Spring Security, WebSocket, async, Jackson, observability wiring), `crypto` (field-level encryption of sensitive columns), and `exception` (domain exceptions shared by every module's service layer).

### Scraping Flow

```
User submits scrape request
        │
        ▼
  Scraping session created (status: IN_PROGRESS)
        │
        ▼
  Background worker polls queue, fetches listings via funda-proxy
        │
        ▼
  Listing module persists/updates listing records
        │
        ▼
  FetchGeocodingCommand → Geocoding consumer
  (Nominatim, rate-limited) stores PostGIS point on listing
        │
        ▼
  FetchPriceHistoryCommand → Price history consumers
  (5 concurrent, max 10/min) fetch and store price data
        │
        ▼
  ScrapingSessionCompleted event published
```

Scraping is fully asynchronous — it never blocks API responses. Sessions are persisted so in-progress work survives restarts. Both geocoding and price history fetching are rate-limited via Guava `RateLimiter` to avoid hammering external services. A `ScrapingTimeoutWatchdog` marks sessions as `TIMED_OUT` if a worker never reports back within 3 minutes.

### Authentication & Authorization

The API is secured as an OAuth2 resource server backed by [Keycloak](https://www.keycloak.org). Every request other than `/actuator/health/**` and the WebSocket handshake (`/ws/chat/**`, which authenticates over STOMP `CONNECT` instead) must carry a valid JWT. Keycloak realm roles are mapped to Spring `ROLE_*` authorities by `SecurityConfig`, and admin-only operations (rescraping a listing, triggering the geocoding backfill, creating/inspecting scraping sessions, manually running an agent task) are enforced with `@PreAuthorize("hasRole('ADMIN')")` at the controller layer. Access-denied responses — both from `@PreAuthorize` and from ownership checks inside services — are rendered as a consistent `ProblemDetail` 403 body.

### Domain Exceptions

Service-layer code never throws Spring MVC's `ResponseStatusException` or Spring Security's `AccessDeniedException` directly — those are HTTP/security-framework types that don't belong outside the web layer. Instead, services throw plain domain exceptions from `com.kropholler.dev.hermes.exception`:

| Exception | HTTP status |
|---|---|
| `NotFoundException` | 404 |
| `UnprocessableEntityException` | 422 |
| `ForbiddenException` | 403 |

`GlobalExceptionHandler` (a `@RestControllerAdvice`) translates each to a `ProblemDetail` response. This keeps controllers and services free of HTTP-specific imports while still producing a uniform error contract for the frontend.

### Field-Level Encryption

Sensitive columns (user profile address/email, chat message content, agent task name/payload, notification title/body) are encrypted at rest via a JPA `AttributeConverter` (`crypto` package). Keys are versioned (`hermes.encryption.keys.<n>` / `hermes.encryption.salts.<n>`) so rotation is possible without downtime: a `ReencryptionRunner` sweeps every module's `Reencryptable` implementation on startup and re-encrypts any row still on an older key version.

### AI Chat

The AI module exposes a WebSocket endpoint (`/ws/chat`) that streams responses from a locally running Ollama LLM. Chat history is persisted per user/session (`ChatHistoryService`) so conversations survive a page reload and are listed in a session sidebar. The backend provides the model with two sets of function-calling tools.

**Listing tools** (`ai.tool`):

| Tool | What it does |
|---|---|
| `ListingSearchTool` | Full-text + filter search over listings |
| `FindPriceDropTool` | Finds listings with the largest price drops |
| `GetListingSummaryTool` | Retrieves the AI-generated summary for a listing |
| `GetPriceHistoryTool` | Returns price history for a listing |
| `CompareListingsTool` | Side-by-side comparison of multiple listings |
| `GetFavouriteListingsTool` | Fetches the user's saved favorites |

**Agent task tools** (`ai.agent.tool`, contributed via `AgentChatToolProvider`):

| Tool | What it does |
|---|---|
| `SaveWatchTool` | Creates a daily watch alert for matching listings |
| `ListWatchesTool` | Lists active watches; optionally cancels one by id |
| `TriggerDigestTool` | Schedules a weekly market digest for one or more cities |
| `TriggerResearchTool` | Queues a one-off background research task |
| `SaveAreaResearchTool` | Queues an area research task for a specific city |

Additional tools can be contributed by implementing the `ChatToolProvider` interface.

### Agent Tasks

Scheduled agent tasks run on a cron-like schedule per user. Supported task types are `WATCH` (daily, tracks specific criteria), `DIGEST` (weekly, per-city summary), `RESEARCH` (one-off), and `AREA_RESEARCH`. A task runs via `AgentTaskScheduler`, executes its handler through `AgentTaskExecutor`, and produces a `Notification` stored in the database and optionally sent by email. Admins can also trigger any task immediately via a "run now" action, independent of its schedule.

### Spatial Search

Listings are geocoded on ingest (via Nominatim) and stored as PostGIS `geometry(Point, 4326)` in the `location` column. The listing search API accepts an optional `radiusKm` parameter; when combined with a `street` + `city` or just `city`, the backend geocodes the search address and filters results using `ST_DWithin`. Users can also save a home address in their profile and search relative to it directly from AI chat, without re-entering an address each time.

### Funda Proxy

A small Python service that sits between the backend and Funda.nl, abstracting the scraping logic away from the JVM.

| Layer | Technology |
|---|---|
| Framework | FastAPI |
| Language | Python 3.12+ |
| Observability | OpenTelemetry (FastAPI instrumentation) |

**Endpoints:**
- `GET /health` — liveness check
- `GET /search?location=...` — search listings by location with optional price/area filters
- `GET /listings/{id}` — fetch a single listing by Funda ID
- `GET /listings/{id}/price-history` — fetch the full price history for a listing

The backend configures the proxy URL via `funda.proxy.url` (defaults to `http://localhost:8001` in dev). Setting `FUNDA_MOCK_MODE=true` (the Docker Compose default) makes the proxy serve fixture data from `mock_client.py`/`mock_fixtures.py` instead of hitting Funda.nl — useful for local development and demos without generating real scraping traffic.

## Getting Started

### Prerequisites

- Java 25+
- Maven 3.9+
- Docker + Docker Compose
- Node.js 20+
- [Ollama](https://ollama.com) (recommended on Windows — see [Ollama setup](#ollama-setup))

### Ollama setup

The AI module uses [Ollama](https://ollama.com) to run `llama3.2:3b` locally for chat and listing summaries.

#### Option 1: Local Ollama (recommended on Windows, especially AMD GPU)

GPU acceleration for Ollama does not work in a Docker Compose container on Windows, including with AMD GPUs (tested on AMD Radeon RX 7900 XT). For GPU-accelerated inference on Windows, install Ollama natively instead.

1. [Download and install Ollama](https://ollama.com/download)
2. Pull the model:
   ```bash
   ollama pull llama3.2:3b
   ```
3. Ollama starts automatically on `http://localhost:11434`. The backend connects to it via `host.docker.internal:11434` from inside Docker.

No extra flags are needed — local Ollama is the default when you run `docker compose up -d`.

#### Option 2: Dockerized Ollama (Linux / CPU-only)

On Linux or when GPU acceleration is not required, you can run Ollama inside Docker Compose using the `ollama` profile. The container will pull the model on first start.

```bash
cd hermes-backend
SPRING_AI_OLLAMA_BASE_URL=http://ollama:11434 docker compose --profile ollama up -d
```

### Run locally

**Start infrastructure (with local Ollama — default):**
```bash
cd hermes-backend
docker compose up -d
```

This also starts Keycloak on `localhost:8081` with a `hermes` realm auto-imported from `keycloak/realm-export.json`, seeded with two test users for local login:

| Username | Password | Role |
|---|---|---|
| `testuser` | `password` | user |
| `testadmin` | `password` | admin |

Field-level encryption requires an encryption key/salt; Compose provides dev-only defaults (`HERMES_ENCRYPTION_KEY_V1` / `HERMES_ENCRYPTION_SALT_V1`) so no extra setup is needed locally — override them with real secrets for any non-local environment.

**Start backend:**
```bash
./mvnw spring-boot:run
```

**Start frontend (development):**
```bash
cd hermes-frontend
npm install
npm start
```

The frontend dev server proxies `/api` to `localhost:8080` and redirects unauthenticated users to Keycloak for login.

**Run backend with Testcontainers (no Docker Compose needed):**
```bash
./mvnw spring-boot:test-run
```

### Run tests

```bash
cd hermes-backend
./mvnw test
```

The test suite has two layers:

- **Unit / slice tests** — JUnit 5 + Mockito covering services, mappers, controllers, and repository queries. Spring Modulith's module structure verification runs here too — module dependency violations fail the build.
- **End-to-end Cucumber tests** — 62 scenarios across 7 feature files (`favorites`, `listing-search`, `listing-report`, `notifications`, `watches`, `profile`, `scraping`) run against a real Spring Boot context with a PostGIS Testcontainer and MockMvc. Each scenario exercises the full HTTP → service → database path, including auth (JWT via `SecurityMockMvcRequestPostProcessors`), ownership checks, and error responses.

Repository/context tests and Cucumber tests require Docker to be running. JaCoCo coverage reports are generated under `target/site/jacoco/`.

## Key Design Decisions

**Modulith over microservices** — a single deployable unit reduces operational complexity while Modulith enforces module boundaries at compile/test time, preventing architectural drift.

**ActiveMQ Artemis for async post-ingest work** — after a listing is persisted, `FetchGeocodingCommand` and `FetchPriceHistoryCommand` are sent to JMS queues. Both are processed by rate-limited consumers, preventing connection pool exhaustion and avoiding hammering external services on bulk scrapes.

**PostGIS for spatial search** — listings are geocoded on ingest via Nominatim and stored as PostGIS geometry. Radius searches use `ST_DWithin` with a geography cast for accurate metre-based distance. The search center is resolved by geocoding the user-supplied address; a city is required alongside a street name to avoid Nominatim resolving the wrong location.

**OpenAPI contract-first** — each module has its own YAML spec under `openapi/`. The Maven build generates typed interfaces (e.g. `ListingsApi`, `FavoritesApi`) from them; controllers implement those interfaces. This keeps specs and implementation in sync without runtime magic.

**JPA Specification for search** — listing search uses Spring Data's `JpaSpecificationExecutor` with dynamically composed predicates, keeping query logic out of the controller and testable in isolation.

**Streaming AI chat via WebSocket** — the chat endpoint streams Reactor `Flux<String>` tokens directly over a STOMP WebSocket connection, sending each token as a `TOKEN` frame as it arrives. A `RESULT` frame with any listing cards is sent after the stream completes.

**Local LLM via Ollama** — all AI inference (chat and summaries) runs on local infrastructure, avoiding external API costs and keeping listing data private.

**Keycloak-issued JWTs, not sessions** — the resource server is fully stateless; every request is authorized from realm roles embedded in the token, keeping the backend horizontally scalable without sticky sessions.

**Domain exceptions over framework exceptions in services** — services throw `NotFoundException`/`ForbiddenException`/`UnprocessableEntityException` rather than Spring MVC's `ResponseStatusException` or Spring Security's `AccessDeniedException`, so the service layer has no compile-time dependency on web or security frameworks. A single `GlobalExceptionHandler` maps them to `ProblemDetail` responses.

**Field-level encryption with versioned keys** — encrypting sensitive columns via a JPA converter keeps the encryption concern out of entity/service code, while per-row key versioning lets keys be rotated by re-encrypting in the background instead of requiring a maintenance window.
