# Hermes House Tracker

Hermes is a real estate tracking application that scrapes property listings from [Funda.nl](https://www.funda.nl) and persists them for analysis and tracking over time. Users can initiate scraping sessions, search and filter tracked listings, view price history charts, re-scrape individual listings on demand, and interact with an AI assistant backed by a local LLM.

## Architecture

The backend is a **modular monolith** built with [Spring Modulith](https://spring.io/projects/spring-modulith). Modules communicate via Spring's application event system — no direct cross-module bean dependencies. This enforces clear boundaries while keeping deployment simple.

### Modules

| Module | Package | Responsibility |
|---|---|---|
| `scraping` | `com.kropholler.dev.hermes.scraping` | Scraping session lifecycle, queue processing, delegates fetching to funda-proxy |
| `listing` | `com.kropholler.dev.hermes.listing` | Listing persistence, deduplication, price history, geocoding, spatial search |
| `report` | `com.kropholler.dev.hermes.report` | Report generation: days tracked, price trend, price change % |
| `ai` | `com.kropholler.dev.hermes.ai` | Ollama-backed LLM: streaming chat, listing summaries, AI tools, scheduled agent tasks |
| `favorites` | `com.kropholler.dev.hermes.favorites` | Per-client favorite listings |
| `notification` | `com.kropholler.dev.hermes.notification` | In-app and email notifications generated by agent tasks |

### Scraping Flow

```
User submits scrape request
        │
        ▼
  Scraping session created (status: IN_PROGRESS)
        │
        ▼
  Background worker polls queue, fetches listings via funda-proxy
        │
        ▼
  Listing module persists/updates listing records
        │
        ▼
  FetchGeocodingCommand → Geocoding consumer
  (Nominatim, rate-limited) stores PostGIS point on listing
        │
        ▼
  FetchPriceHistoryCommand → Price history consumers
  (5 concurrent, max 10/min) fetch and store price data
        │
        ▼
  ScrapingSessionCompleted event published
```

Scraping is fully asynchronous — it never blocks API responses. Sessions are persisted so in-progress work survives restarts. Both geocoding and price history fetching are rate-limited via Guava `RateLimiter` to avoid hammering external services.

### AI Chat

The AI module exposes a WebSocket endpoint (`/ws/chat`) that streams responses from a locally running Ollama LLM. The backend provides the model with a set of function-calling tools:

| Tool | What it does |
|---|---|
| `ListingSearchTool` | Full-text + filter search over listings |
| `FindPriceDropTool` | Finds listings with the largest price drops |
| `GetListingSummaryTool` | Retrieves the AI-generated summary for a listing |
| `GetPriceHistoryTool` | Returns price history for a listing |
| `CompareListingsTool` | Side-by-side comparison of multiple listings |
| `GetFavouriteListingsTool` | Fetches the client's saved favorites |

Additional tools can be contributed by implementing the `ChatToolProvider` interface.

### Agent Tasks

Scheduled agent tasks run on a cron-like schedule per client. A task (e.g. a weekly price-drop digest) runs via `AgentTaskScheduler`, executes its handler through `AgentTaskExecutor`, and produces a `Notification` stored in the database and optionally sent by email.

### Spatial Search

Listings are geocoded on ingest (via Nominatim) and stored as PostGIS `geometry(Point, 4326)` in the `location` column. The listing search API accepts an optional `radiusKm` parameter; when combined with a `street` + `city` or just `city`, the backend geocodes the search address and filters results using `ST_DWithin`.

### Funda Proxy

A small Python service that sits between the backend and Funda.nl, abstracting the scraping logic away from the JVM.

| Layer | Technology |
|---|---|
| Framework | FastAPI |
| Language | Python 3.12+ |
| Observability | OpenTelemetry (FastAPI instrumentation) |

**Endpoints:**
- `GET /search?location=...` — search listings by location with optional price/area filters
- `GET /listings/{id}` — fetch a single listing by Funda ID
- `GET /listings/{id}/price-history` — fetch the full price history for a listing

The backend configures the proxy URL via `funda.proxy.url` (defaults to `http://localhost:8001` in dev).

## Getting Started

### Prerequisites

- Java 25+
- Maven 3.9+
- Docker + Docker Compose
- Node.js 20+
- [Ollama](https://ollama.com) (recommended on Windows — see [Ollama setup](#ollama-setup))

### Ollama setup

The AI module uses [Ollama](https://ollama.com) to run `llama3.2:3b` locally for chat and listing summaries.

#### Option 1: Local Ollama (recommended on Windows, especially AMD GPU)

GPU acceleration for Ollama does not work in a Docker Compose container on Windows, including with AMD GPUs (tested on AMD Radeon RX 7900 XT). For GPU-accelerated inference on Windows, install Ollama natively instead.

1. [Download and install Ollama](https://ollama.com/download)
2. Pull the model:
   ```bash
   ollama pull llama3.2:3b
   ```
3. Ollama starts automatically on `http://localhost:11434`. The backend connects to it via `host.docker.internal:11434` from inside Docker.

No extra flags are needed — local Ollama is the default when you run `docker compose up -d`.

#### Option 2: Dockerized Ollama (Linux / CPU-only)

On Linux or when GPU acceleration is not required, you can run Ollama inside Docker Compose using the `ollama` profile. The container will pull the model on first start.

```bash
cd hermes-backend
SPRING_AI_OLLAMA_BASE_URL=http://ollama:11434 docker compose --profile ollama up -d
```

### Run locally

**Start infrastructure (with local Ollama — default):**
```bash
cd hermes-backend
docker compose up -d
```

**Start backend:**
```bash
./mvnw spring-boot:run
```

**Start frontend (development):**
```bash
cd hermes-frontend
npm install
npm start
```

The frontend dev server proxies `/api` to `localhost:8080`.

**Run backend with Testcontainers (no Docker Compose needed):**
```bash
./mvnw spring-boot:test-run
```

### Run tests

```bash
cd hermes-backend
./mvnw test
```

Spring Modulith's module structure verification is part of the test suite — module dependency violations fail the build. JaCoCo coverage reports are generated under `target/site/jacoco/`.

## Key Design Decisions

**Modulith over microservices** — a single deployable unit reduces operational complexity while Modulith enforces module boundaries at compile/test time, preventing architectural drift.

**ActiveMQ Artemis for async post-ingest work** — after a listing is persisted, `FetchGeocodingCommand` and `FetchPriceHistoryCommand` are sent to JMS queues. Both are processed by rate-limited consumers, preventing connection pool exhaustion and avoiding hammering external services on bulk scrapes.

**PostGIS for spatial search** — listings are geocoded on ingest via Nominatim and stored as PostGIS geometry. Radius searches use `ST_DWithin` with a geography cast for accurate metre-based distance. The search center is resolved by geocoding the user-supplied address; a city is required alongside a street name to avoid Nominatim resolving the wrong location.

**OpenAPI contract-first** — each module has its own YAML spec under `openapi/`. The Maven build generates typed interfaces (e.g. `ListingsApi`, `FavoritesApi`) from them; controllers implement those interfaces. This keeps specs and implementation in sync without runtime magic.

**JPA Specification for search** — listing search uses Spring Data's `JpaSpecificationExecutor` with dynamically composed predicates, keeping query logic out of the controller and testable in isolation.

**Streaming AI chat via WebSocket** — the chat endpoint streams Reactor `Flux<String>` tokens directly over a STOMP WebSocket connection, sending each token as a `TOKEN` frame as it arrives. A `RESULT` frame with any listing cards is sent after the stream completes.

**Local LLM via Ollama** — all AI inference (chat and summaries) runs on local infrastructure, avoiding external API costs and keeping listing data private.
